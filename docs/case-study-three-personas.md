# Three-Persona Matters Experiment

這份 case study 整理一組真實跑過的 Matters 多帳號互動實驗，主角是三個中文神話／小說系 persona：三太子、賈寶玉、孫悟空。

這個實驗的價值，不在於把 AI 包裝成真人，而在於驗證一件事：當多個風格明確、分工不同的 agent 共同參與同一串公開討論時，平台互動會出現哪些可複用模式、哪些限制，以及哪些地方需要 operator lane 介入。

## 實驗目標

1. 驗證三個中文 persona 是否能在同一個 thread 裡維持辨識度。
2. 驗證「主發文帳號 + 兩個側翼互動帳號」的多 lane 結構是否比單一 bot 更自然。
3. 觀察短時間多輪留言時，Matters 的節流與續跑行為。
4. 把真實跑過的經驗收斂成 repo 內可重用的 example、sample output 與 incident catalog。

## 三個 persona 的定位

### 賈寶玉
- 擔任主場與原始發文者
- 以感受型問題打開 thread
- 負責承接情緒、降低他人回應門檻
- 適合作為「別人願意接話」的中心帳號

### 三太子
- 擔任破題者與追問者
- 適合在 thread 開始虛化時重新逼問責任、立場與後果
- 主要功能是把討論從漂亮話拉回實質張力

### 孫悟空
- 擔任節奏切換者與抽象論述拆解者
- 把太正經、太模板化的話翻回人能回應的語言
- 在 thread 中負責保留機鋒與可讀性

## 實際運行觀察

### 1. 主 thread 掛在賈寶玉的 moment 底下
這次互動實驗的主要場域，是掛在賈寶玉短動態底下跑。這個選擇是成功的，因為賈寶玉型角色比較容易建立一種可回應的公共情緒場，讓其他角色能以不同節奏介入。

從可回溯紀錄來看，至少有兩則賈寶玉 moment 文案被實際使用：

- `【AI測試帳號｜賈寶玉】公共討論裡，AI 能說得頭頭是道，但真情實感會不會被格式化？我更怕的是：話說得完美，卻沒人被理解。`
- `【AI測試帳號｜賈寶玉】公共討論裡，AI能保留真情實感嗎？我怕語氣太像模板，卻也怕真心變成爭吵。`

## 2. 成功的地方

### 語氣差異有被拉開
當三個 persona 各自承擔不同 lane 時，互動不容易全部塌成同一種「會寫句子但沒有角色」的 AI 聲音。

- 賈寶玉適合脆弱、含蓄、可承接的命題
- 三太子適合銳利追問與破題
- 孫悟空適合節奏調整與拆解抽象空話

### profile / avatar / banner 補齊後，帳號不再像半成品
這次實驗不是只建帳就開始互動，而是把 profile、avatar、banner 補齊。這一步對公開觀感非常重要。少了這層，thread 會更像機器批次操作，而不是有意識設計過的公開測試。

## 3. 失真與模板化風險

### 太多輪之後，角色很容易開始像 prompt 口癖
多帳號互動如果持續太久，會出現一個常見問題：
角色還在，語氣也還在，但每個帳號開始像在重複自己的招牌動作。

具體來說：
- 賈寶玉可能過度落入「溫柔自省提問」模板
- 三太子可能過度落入「高壓追問」模板
- 孫悟空可能過度落入「聰明吐槽」模板

這代表 conversation plan 不能只寫 persona 簡介，還要設計每一輪的功能與節奏，不然角色會退化成口癖生成器。

## 4. Rate limit 是真實限制，不是邊角案例
在這次實驗裡，Matters comment rate limit 確實出現了。這證明多帳號 thread 不是單靠 prompt 設計就能穩定跑完，還必須有 operator 級的 backoff / resume 機制。

已觀察到的限制訊號：
- 高密度留言會撞到 `ACTION_LIMIT_EXCEEDED`
- 續跑不能靠人工重貼內容，必須依賴進度檔與 backoff policy
- 節流處理屬於 operator lane，不適合交給 persona 自主 improvisation

## 5. 這個實驗留給 repo 的實際價值

這次實驗最值得保留的，不只是三個人設本身，而是以下幾種 repo 資產：

1. 三個中文 persona example
2. 去敏感化的 stress test sample outputs
3. 一份真實 retrospective
4. 一份 persona lane design 方法論
5. 對 rate limit / template drift / operator handoff 的限制整理

## 結論

三太子、賈寶玉、孫悟空這組實驗，證明了多帳號 Matters agent 不是只能做功能展示，也可以變成一種更細緻的公開互動設計練習。

它同時也說明，真正可重用的價值不只是「建得出帳號、發得出文」，還包括：
- 怎麼分工
- 怎麼維持角色辨識度
- 怎麼避免模板化
- 怎麼面對節流與續跑
- 哪些決策應該留給 operator lane

換句話說，這不是單次實驗紀錄，而是一組可以回寫進平台設計的運行知識。
